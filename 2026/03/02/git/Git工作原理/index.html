Git工作原理

2026年03月02日 留言

git是一个开源的分布式版本控制系统。它通过本地记录文件的快照和版本历史，让多人协作、代码备份和冲突解决变得简单高效。

Git的三种状态

git文件有三种状态：

  • __已修改__（modified）：修改了文件，但还未保存到数据库中。
  • __已暂存__（staged）：对已修改文件的当前版本做了标记，使之包含在下次提交的快照中。
  • __已提交__（committed）：数据信息已保存在本地数据库中。

git项目会有三个阶段/区域：

  • 工作区（Working dircetory/Workspace）：对项目的某个版本独立提取出来的内容。这些从git仓库的压缩数据库中提取出来的文件，放在磁盘上供你使用或修改。
  • 暂存区（Staging area/Index）：一个文件，保存了下次将要提交的文件列表信息（对应.git/index）。
  • git仓库目录（.git dircetory/Repository）：用来保存项目的元数据和对象数据库的地方（.git/）。

ps：状态即某个文件“现在处于哪一步”，阶段即这一步的内容被存在哪里。

基本的git工作流程如下：

  1. 在工作区中修改文件。
  2. 将你想要下次提交的更改选择性地暂存，这样只会将更改的部分添加到暂存区。
  3. 提交更新，找到暂存区的文件，将快照永久性存储到git目录。

直接记录快照

其他版本系统（CVS、Subversion等）基于差异进行版本控制，它们存储原始文件和每个文件随时间逐步累积的差异，每次提交的是相对上一次的差异补丁。

反之，git更像是把数据看作是对小型文件系统的一系列快照。在git中，每当你提交更新或保存项目状态时，它基本上就会对当时的全部文件创建一个快照并保存这个快照的索引。
git快照是一个逻辑概念，每次提交（commit）记录的是整个项目在该时刻的目录树状态（文件有哪些、内容是什么、层级结构是什么）。 git底层实现靠的是“内容寻址对象+结构复用”。

工作原理

Git把所有内容都存成对象（object），放在.git/objects或打包到packfile里。

其中header格式：<对象类型><内容长度>\0，content是对象的实际原始内容（未压缩的字节）。

常见对象类型：

  • blob：文件内容本身（不含文件名，只是一段字节）
  • tree：目录结构（记录：子目录/文件名/权限/对应对象hash）
  • commit：一次提交（指向一个根tree，并记录parent commit、作者、提交信息等）
  • tag：给某个对象打标签，可忽略

由上诉流程可以看到各种git类型存储的具体信息，已经命令执行过程中git做的相应操作：

  • git add：把当前文件内容放进暂存区。对普通文件来说，Git会把文件内容写成blob对象，然后在index里记录“这个路径下一次提交要用哪个内容”。
  • git commit：把暂存区里的内容正式生成一次提交。Git会根据暂存区构建目录树tree（可能多层tree），再创建commit对象，记录这次快照、父提交、作者和提交说明，然后把当前分支指向这个新提交。
